1. Purpose of This Note
This note is meant to help the Google Partner / implementation partner align their technical scope with the intended HKHT AI for Seva architecture for WF-002 Voice Agent + CRM Follow-up.
This is not a rejection of the partner’s current work.
This is not adding random new scope.
This is a clarification of the intended architecture so the partner can classify, phase and strengthen the implementation plan correctly.
The goal is to make early execution small but production-shaped, platform-compatible and ready for 30-day production base, 60-day stabilization and 75-day enterprise-scale readiness.

2. Core Understanding Expected from the Partner
WF-002 is not only a call transcript summarizer and not only a bot demo.
It is the voice-and-follow-up conversion module of the HKHT AI Revenue & Seva Growth Platform.
It should convert approved contacts into structured relationship history, next actions, callbacks, WhatsApp follow-up, human escalation and dashboard visibility.
It must preserve common IDs, source tracking, approvals, CRM/ERP readiness, audit trail and future scale.
It must be connected to WF-001 Yatra, WF-003 Digital Marketing, WF-006 CRM/DBMS, WF-007 Donor/LLP/CSR, WF-008 task governance, WF-009 finance/DCC references and WF-014 Community Outreach.
Evidence basis: Latest Document B sections 1, 4, 6, 11 and Cross-Workflow Integration; GE Gap Analysis WF-002 section.

5. Scope Alignment with HKHT GE Technical Scope vs Gap Analysis
The HKHT GE Technical Scope positions WF-002 as a light custom / P2 workflow where the outsourced dialer provides transcripts and GE can process post-call intent and next steps. That is useful, but the Gap Analysis clarifies that WF-002 must not stop at summarizing transcripts.
Practical point | WF-002 implication
What is covered | Post-call transcript ingestion, cleaning, intent matching, next-step extraction and CRM writeback concept.
What is missing or unclear | Whether every call becomes persistent CRM relationship history; whether next actions trigger WhatsApp/callback/human escalation; whether campaign source stays attached; whether failed/unresolved calls remain visible.
What needs classification | Live voice/dialer, CRM API, WhatsApp API, dashboard, BigQuery/Looker, ClickUp task integration, Vertex/Agent Builder and external vendor components.
What may be mock/demo only early | Full CRM API writeback, WhatsApp send, payment/DCC linkage, predictive scoring and multi-center packaging unless dependencies are confirmed.
What needs HKHT data/access | Contact batch, CRM schema, transcript export, script/FAQ, outcome taxonomy, approved templates, owner list and dashboard KPIs.
What needs stronger production alignment | Persistent IDs, source tracking, CRM timeline, data quality checks, fallback procedure, golden journey QA and leadership command center contribution.

Evidence basis: HKHT GE Technical Scope; HKHT GE Technical Scope — Gap Analysis.

7. What the Partner Should Not Miss
Common IDs: Contact ID, Donor ID, Yatri ID, Family ID, Campaign ID, Event ID, Call ID, Task ID and Owner ID where applicable.
CRM/DBMS remains the relationship source of truth for calls, follow-ups, source, opt-outs, tags and relationship stage.
ERP-ready fields must be captured without forcing unstable early ERP integration.
Human approval gates are mandatory for donor-sensitive, HNI/CSR/LLP, finance/payment/receipt, public and devotional/philosophical content.
Audit logs should show what AI did, what human approved and what was written back.
Role ownership must be validated; transcript-derived names should not be treated as final.
Fallback process must exist for voice, CRM, WhatsApp and dashboard failure.
API registry must classify Hello Leads, Tata Tele, voice vendor, WhatsApp, payment/DCC, ERP and ClickUp/task APIs.
Data quality checks must run before calling.
Golden journey testing must validate the full journey, not only one screen.
KCKE and Media AI boundaries must remain clear.
WF-002 must contribute to the leadership command center and future WF-013 packaging.
